# Отказоустойчивость и переключение трафика
Основные мысли, представим у нас 2 цода:
- Отказывается один из цодов
- Происходит переключение трафика
- В первом цоде у нас начинает сильно расти нагрузка на имеющиеся реплики
- Реплики могут стать недоступны ввиду таймаутов, залипании хелсчеков. Они могут начать рестартиться, если у сервисов нет выделенного потока ответственного за обработку хелсчеков
- В нормальной ситуации, я считаю недостаточность ресурсов под нагрузкой не должна приводить к рестарту сервиса - только если это утечка памяти, которую можно попробовать диагностировать автоматически (например, логика обработки пробы включает аналитические показатели по памяти (количество объектов и тд, что нехарактерно - какие то симптомы))
- Соответственно лавина трафика может замедлить сервисы, тут зависит от специфики
- Соответственно если у нас сервисы не умеют хорошо приспосабливаться, то тут должен быть механизм какого то аварийного рейт лимита в зависимости от количества реплик. Реплики могут начать подниматься по HPA либо при недоступности цод встанут. И порог аварийного рейт лимита повыситься.
- То есть между количеством реплик, и пускаемым на реплики трафиком (балансировщиком к примеру) должна быть какая то обратная связь

When one datacenter (DC) fails, the remaining DC receives 100% of the traffic but only has 50% (or less) of the capacity. Standard HPA is indeed too slow (usually minutes) to prevent a cascade failure during this spike. <br>
То есть ресурс - это термин Capacity здесь можно использовать <br>
To implement Backpressure, you need a layered approach. You cannot rely on just one mechanism. You need to shed load (reject traffic) immediately while scaling happens in the background.

## The Strategy: 3 Layers of Defense
- Layer 1: Load Shedding (Immediate): The application or sidecar rejects excess traffic before resources (CPU/Memory) are exhausted. This returns 503 Service Unavailable with a Retry-After header.
- Layer 2: Concurrency Limiting (Infrastructure): The Ingress Controller or Service Mesh limits the number of simultaneous requests hitting the pods.
- Layer 3: Predictive Scaling (Fast Reaction): Replace/augment HPA with KEDA to scale based on request rate (RPS) rather than CPU usage, which is faster.

Что мне тут видится: 
- Сделать оператор, который может ходить на кубовые (ингресс, gateway-api) и внешние балансировщики и проставлять rate-limit в зависимости от количества реплик. Нагрузка (rps) высчивается за счёт нагрузочного тестирования (количество rps/одна реплика например КОЭФФИЦИЕНТ)
- Балансировщик должен каким либо способом предоставлять метрику текущего rps поступающего на балансер (API запросы, Prometheus метрика и тд)
- В случае на попадания под rate limit, балансировщики ведут на страничку (попробуйте через 10 секунд)
- Оператор видит превышение, скейлит деплойменты и поднимает rate limit в соответствие с нашим коэффициентом

### Implementation 1: Application-Level Backpressure (The "Signal")
Вот один из вариантов это учитывать на уровне сервиса. Мне кажется сервис таким заниматься не должен, если он может отдавать метрики необходимые для инфраструктуры и отвечает всем требованиям среды Kubernetes, умеет масштабироваться и линейно выдерживать нагрузку
```
const MAX_CONCURRENT_REQUESTS = 50;
let currentRequests = 0;

function backpressureMiddleware(req, res, next) {
    if (currentRequests >= MAX_CONCURRENT_REQUESTS) {
        // SIGNAL BACKPRESSURE
        res.set('Retry-After', '5'); // Tell LB/Client to wait 5 seconds
        return res.status(503).json({ error: 'Service Overloaded', retry: 5 });
    }

    currentRequests++;
    res.on('finish', () => {
        currentRequests--;
    });
    
    next();
}

app.use(backpressureMiddleware);
```
### Implementation 2: Infrastructure Backpressure (Envoy / Istio / Nginx)
Адаптивная concurrency. Интересный вариант, снижается количество успешных запросов - понижается concurrency лимит <br>
Даже адаптироваться под рост latency умеет - Istio (based on Envoy) has a filter specifically for this. It dynamically adjusts the concurrency limit based on latency. If latency spikes (indicating overload), it reduces the allowed concurrency.
```
apiVersion: networking.istio.io/v1alpha3
kind: EnvoyFilter
metadata:
  name: adaptive-concurrency
  namespace: production
spec:
  workloadSelector:
    labels:
      app: my-microservice
  configPatches:
  - applyTo: HTTP_FILTER
    match:
      context: SIDECAR_INBOUND
      listener:
        filterChain:
          filter:
            name: "envoy.filters.network.http_connection_manager"
            subFilter:
              name: "envoy.filters.http.router"
    patch:
      operation: INSERT_BEFORE
      value:
        name: envoy.filters.http.adaptive_concurrency
        typed_config:
          "@type": type.googleapis.com/udpa.type.v1.TypedStruct
          type_url: type.googleapis.com/envoy.extensions.filters.http.adaptive_concurrency.v3.AdaptiveConcurrency
          value:
            min_concurrency_limit: 10
            max_concurrency_limit: 100
            gradient_controller_config:
              sample_aggregate_percent: 10
              success_rate_threshold: 0.95
```
How it works: If the success rate drops below 95% or latency increases, Envoy automatically lowers the concurrency limit, rejecting excess traffic at the proxy level with 503s. <br>
У nginx чё то есть но херня <br>
If you use Nginx Ingress, you can limit connections per key.
```

    nginx.ingress.kubernetes.io/limit-connections: "20"
    nginx.ingress.kubernetes.io/limit-rps: "50"
```
